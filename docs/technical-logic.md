# Neuron Script 项目技术逻辑文档

## 1. 项目整体技术逻辑

Neuron Script 采用 `FastAPI + 原生 HTML/CSS/JavaScript + DeepSeek + Supabase` 的技术架构，围绕“设定输入 -> 角色构建 -> 大纲规划 -> 正文撰写 -> 收藏导出 -> 账户沉淀”形成完整的人机协同创作闭环。

前端负责参数采集、角色关系图可视化、历史版本管理、局部编辑和结果展示；后端负责提示词拼装、LLM 调用、异步任务管理、JSON 结果校验、导出文件生成；Supabase 负责登录鉴权、游客模式、账户信息和后续云端数据沉淀。系统强调“人主导、AI协作”，因此所有生成结果都允许用户继续编辑、暂停、恢复、放弃、导出或迭代。

## 2. 提示词工程

### 2.1 结构化 JSON 输出

系统的核心生成任务，包括角色关系补充、大纲生成与正文生成，均要求模型返回严格 JSON，而非自然语言段落。这样设计有三个目的：一是让大语言模型在字段级别理解任务目标，减少“答非所问”；二是便于后端用 Pydantic 直接校验字段合法性；三是前端可以直接将返回结果映射到角色卡、章节列表、正文面板等组件，降低二次解析成本。

后端在调用模型时统一启用 `response_format=json_object`，同时在 system prompt 中重复强调“只能返回一个可被 `json.loads` 解析的 JSON 对象”。若模型输出存在截断、尾逗号、中文符号、字符串双引号未转义等问题，`llm_client.py` 会执行自动修复、结构补全与二次重试，尽可能把模型输出转为可用结果。

### 2.2 提示词分层设计

提示词采用“公共风格指导 + 任务专属 prompt”双层结构。

- 公共层负责约束题材逻辑与语言风格，保证角色命名、关系设计、大纲风格与正文措辞在同一审美基调下保持一致。
- 任务层分别面向“角色起名”“角色关系补充”“大纲生成”“正文生成”“正文局部重写”五类任务，避免不同任务混用提示词导致输出目标模糊。

这种拆分方式使系统既能复用统一创作规则，又能针对不同任务施加差异化约束。

### 2.3 角色起名逻辑

系统支持在大纲生成前自动为未命名角色补全姓名。角色命名不是随机生成，而是基于权重进行综合判断：

- 第一优先级是世界观的社会背景与时间背景，用于决定命名的时代感、文化语境与语言习惯。
- 第二优先级是故事梗概，用于判断角色在剧情中的功能定位与象征意义。
- 第三优先级是已有角色姓名，用于保证整部作品命名风格统一，避免人名气质割裂。
- 第四优先级是角色信息与角色关系，用于体现性格、动机、命运与人物间的呼应、对立或映衬。

在生成结果进入系统前，后端还会做二次校验，包括：是否为空、是否与已有角色重名、是否包含解释性文本、是否夹带引号或编号。若不合规则要求模型重新生成，最多重试三次。

### 2.4 大纲生成逻辑

大纲生成采用“四段式叙事结构 + 逐章细纲”的方式。系统会先根据总字数和单章字数自动推算章节数，再将章节分配到“开端-发展-高潮-结局”四段结构中，其中发展段权重最高，高潮次之，结局最短。该权重设计有助于避免大纲前重后轻或高潮不足的问题。

在提示词中，系统明确要求模型同时考虑以下因素：

- 草蛇灰线与伏笔回收
- 人物弧光与阶段性微变化
- 事件推动而非生硬说理
- 角色关系变化对情节推进的作用
- 每章概要、关键事件、章末悬念的可写性

同时，为防止响应过长导致截断，系统会对 `summary`、`key_events`、`cliffhanger` 等字段设置字数上限，并在 JSON 被截断时触发短文本重生成策略。

### 2.5 正文生成逻辑

正文生成以“当前章节细纲 + 全局大纲 + 已生成章节摘要 + 故事设定 + 角色关系”为输入。系统采用逐章串行生成，而不是一口气生成全书，原因在于：

- 逐章生成更利于维护上下文连续性
- 能将已完成章节摘要作为后续章节的连续性记忆
- 便于用户在中途暂停、修正大纲或局部返工

正文提示词重点融合了冰山理论、人物描写方法、景物描写、修辞手法、多视角叙事、伏笔回收等写作理论，但这些规则最终服务于“可读的文学文本”，而不是让模型机械完成格式任务。

### 2.6 上下游任务路由

系统的 LLM 任务并不是孤立执行，而是按创作链路逐层传递上下文：

- 基本信息与世界观设定进入角色命名、关系补充和大纲生成
- 角色卡与角色关系进入大纲生成和正文生成
- 已确认大纲进入正文生成与局部重写
- 已生成章节摘要进入后续章节生成，保证连续性

因此，系统本质上是一个“有状态的创作管线”，而非单次问答式调用。任何前置设定发生变更时，后续产物都需要视情况清空或重新生成，以避免上下文污染。

## 3. 核心功能模块技术逻辑

### 3.1 创作中心：基本信息

技术逻辑：

基本信息模块承担全局创作输入的汇总职责，字段包含故事类型、故事梗概、语言风格、世界观时间背景、物理环境、社会环境、总字数、单章字数等。后端以 `StoryDraftRequest` 作为统一数据模型接收这些输入，并对必填项、数值范围和空值做校验与标准化处理。

为降低用户一次性填写负担，系统仅强制要求梗概和篇幅核心参数，其余信息允许为空；当用户未填写完整信息时，模型可在大纲生成中进行有限的合理补全，并把补全部分写入 `inferred_details`，保证“信息缺失但创作不断流”。

前端会将用户当前填写内容、历史版本、当前阶段、图谱视角、已生成大纲和正文一并保存为本地工作区快照，支持刷新后恢复与中途退出续写。

### 3.2 角色关系：角色档案卡

技术逻辑：

角色档案卡本质上是结构化角色数据库的前端视图。每张卡片对应一个 `CharacterCard` 数据对象，包含姓名、性别、年龄、职业、国籍、性格、外貌、价值观、核心动机以及图谱坐标等字段。字段设计覆盖“身份信息 + 外在形象 + 内在驱动 + 图谱定位”四个维度，既服务于用户思考，也服务于模型理解人物。

后端在真正调用模型前会先清洗角色数据，例如去除空白字符、统一命名格式、同步关系中的角色展示名，避免因为前端输入随意导致提示词上下文不稳定。

### 3.3 角色关系网：手动编辑角色关系

技术逻辑：

角色关系网将每位角色渲染为带姓名的椭圆节点，关系以 `source_id -> target_id` 的有向边表示，边上显示关系标签。系统使用角色 `id` 作为真实关联键，而不是直接用角色名关联，以避免用户改名后关系断裂。

用户创建关系时，前端记录起点角色、终点角色、关系名和单双向状态，并生成一条 `CharacterRelation` 记录。若是双向关系，前端和后端都以“两个方向分别存储、同时标记 `bidirectional=true`”的方式处理，从而兼容可视化绘制与模型理解。

角色增删改后，关系网会同步刷新节点与边；图谱同时保留拖拽、缩放、视口平移等状态，增强长篇作品下的编辑可用性。

### 3.4 AI 补充关系

技术逻辑：

AI 补充关系模块读取用户已保存的梗概、角色卡和已有关系，调用专门的 `relation_supplement_prompt` 生成“尚未填写但有助于剧情推进”的关系。系统严格要求 AI 只补充空白关系，不得覆盖、改写或重复输出已有关系。

为了避免 AI 为了“填满关系图”而过度生成，提示词中额外限制：

- 关系补充必须服务于剧情推进
- 允许留白，不强制补满
- 优先补充关键关系而非边缘噪音关系

后端会对返回的关系做格式校验、去重和合法性筛查，再合并到当前关系网中，并将 AI 补充关系与用户手动关系通过 `relation_source` 字段区分。

### 3.5 停止生成

技术逻辑：

“停止生成”不是简单中断前端动画，而是依托后端的异步任务管理器实现。角色关系补充、大纲生成、正文生成均可作为独立 LLM 任务提交到 `LlmTaskManager`。任务对象包含 `running / paused / completed / failed / discarded` 五种状态，以及任务类型、时间戳、结果和错误信息。

当用户点击“停止生成”时，前端调用 `/api/llm-tasks/{id}/pause`，后端取消当前协程任务并将状态置为 `paused`。后续用户可以：

- 继续生成：系统根据暂停前保存的输入重新发起同一任务
- 放弃生成：系统将任务置为 `discarded`，释放当前结果占位

这种设计既节省无效推理开销，也避免误触造成成果完全丢失。

### 3.6 大纲规划：大纲生成与修改

技术逻辑：

大纲生成时，系统会先对故事设定做标准化，再自动为未命名角色补名，随后根据总字数与单章目标字数推导章节规划，最后将故事设定、角色卡、角色关系、风格约束与章节目标打包成结构化 prompt 提交给模型。

大纲修改支持两种路径：

- 全局重生成：保留当前设定，在上一版大纲基础上叠加用户反馈重新生成
- 局部人工修改：用户直接修改章节范围、阶段划分或章节内容

每次生成或重生成后，前端都会自动保存一版大纲历史记录，便于对比版本与回滚。一旦大纲发生变化，系统会清空旧正文，防止大纲和正文版本错位。

### 3.7 正文撰写：正文生成与修改

技术逻辑：

正文生成建立在“锁定大纲”的前提下。系统会遍历所有章节，按顺序将章节标题、概要、关键事件、章末悬念、当前所处阶段、阶段范围以及前文摘要一并送入模型。每完成一章，就把该章摘要加入连续性上下文，供下一章使用。

正文修改支持两种机制：

- 手动编辑：用户直接在文本中修改内容，前端以深蓝色字体和浅蓝色高亮标识人工改动区域，增强人机协作可见性。
- 局部 AI 重写：用户选中文本后，系统把“选中片段 + 上下文 + 当前章信息 + 全局大纲 + 修改指令”提交给模型，只要求返回替换片段本身，从而把改写范围精确控制在局部。

这种方式避免整章重写造成的不确定性扩散，也能强化用户对作品的最终主导权。

### 3.8 文本管理：正文句子收藏

技术逻辑：

收藏功能以“正文中的局部文本片段”为最小单位。用户选中喜欢的句子后，前端会记录句子内容、所属小说标题、章节号、章节标题、收藏时间等元信息，并在正文中为已收藏句子添加珊瑚红波浪下划线，帮助用户建立审美反馈。

当前项目前端已支持收藏、取消收藏、分组展示与本地持久化；在接入 Supabase 后，该收藏结构还可同步写入 `saved_sentences` 表，形成跨设备的个人语料库。

### 3.9 导出功能

技术逻辑：

导出模块统一通过 `/api/export/docx` 接口生成 Word 文档。后端不依赖重量级办公库，而是直接按 OpenXML 最小结构拼装 `.docx` 文件，优点是依赖少、导出快、部署轻量。

导出内容可分为四类：

- 导出基本信息
- 导出角色档案与角色关系
- 导出大纲或单章正文
- 导出全部设定与整本正文

前端在导出前会先将当前页面状态格式化为纯文本，再传给后端统一封装为 Word 文档，因此导出逻辑与页面展示逻辑解耦，便于后续扩展 PDF、Markdown 等新格式。

## 4. 用户中心与数据模块

### 4.1 用户中心：收藏句子管理

技术逻辑：

用户中心将收藏句子按“小说名 -> 章节”进行分组展示，既提高查找效率，也让收藏不只是零散金句，而是与创作语境绑定的灵感资料。用户删除收藏时，系统同步更新收藏列表和正文中的收藏态，保持显示一致性。

当前前端已具备完整的收藏管理视图；接入云端后，收藏列表可通过用户身份隔离，实现真正意义上的个人文学空间。

### 4.2 数据存储

技术逻辑：

系统的数据存储分为本地工作区缓存与云端数据库方案两层：

- 本地层用于快速保存创作现场，包括基本信息、角色卡、关系网、大纲、正文、历史版本、收藏片段、当前阶段与图谱视角等，保障用户刷新页面后仍可继续编辑。
- 云端层用于保存账户级数据，包括用户资料、小说草稿、收藏句子等，支撑跨设备访问与长期沉淀。当前项目已完成鉴权接入与表结构设计，工作区内容的全量云同步仍属于后续扩展方向。

这种“双层存储”设计兼顾了交互流畅性与数据安全性。

### 4.3 用户管理

技术逻辑：

项目当前采用 Supabase Auth 进行用户管理，支持邮箱注册登录、密码登录、密码重置，并已预留游客匿名登录能力。其逻辑分为三层：

- 身份获取：前端通过 `/api/public-config` 获取 Supabase 公共配置并初始化客户端
- 会话维持：使用 Supabase SDK 自动持久化 session 与刷新 token
- 页面路由：根据是否登录决定进入登录页、创作页或用户面板

游客模式适合降低首次使用门槛，正式账户适合承接长期创作记录，两者共同服务于“先体验、再沉淀”的产品路径。

### 4.4 数据库设计

技术逻辑：

当前项目的数据库方案以 Supabase PostgreSQL 为基础，已完成表结构设计文档与鉴权策略规划，核心表包括：

- `profiles`：存储用户扩展信息，如昵称、头像、创建时间
- `novels`：存储小说标题、题材、梗概、世界观、正文、字数和状态
- `saved_sentences`：存储用户收藏的句子及其所属小说

数据库层启用了 Row Level Security，保证用户只能访问自己的数据。这样既满足基础的数据隔离，也为后续多用户协同、作品公开广场、版本回溯等功能留下扩展空间。

### 4.5 验证码与安全校验

技术逻辑：

当前项目已接入 Supabase 邮箱注册与邮箱验证机制，注册后可通过邮件确认身份，忘记密码时也通过邮箱重置链接完成校验。从广义上说，这一流程已经承担了验证码的核心功能，即“确认操作者对邮箱或身份的真实控制权”。

若后续需要补齐更显式的验证码模块，可沿以下方向扩展：

- 邮箱验证码：注册、登录、找回密码时发送一次性验证码
- 短信验证码：用于手机号登录或高风险操作验证
- 图形验证码或行为验证码：用于防刷、防机器注册、防恶意调用生成接口

其中，验证码校验结果应优先绑定到“身份验证”和“生成频控”两个环节，而不是单纯作为前端表单装饰。

## 5. 建议补充的系统级能力

除现有能力外，项目还适合继续补充以下技术逻辑，以提升完整度：

- 生成频控与配额：按游客、普通用户、会员区分每日生成次数、最大字数与并发任务数
- 敏感词与内容安全：在梗概输入、正文导出和分享前增加内容审核
- 版本比对：对大纲版本、正文版本提供差异对比视图
- 章节级保存：允许只重生成单章，而非每次重跑全文
- 角色一致性检查：在生成正文后自动检查称谓、年龄、关系、设定是否前后矛盾

## 6. 交互步骤（面向用户）

用户使用 Neuron Script 创作小说时，共分为四个步骤。

第一步，填写故事基本信息。用户至少需要填写故事类型和故事梗概，再根据需要补充语言风格、世界观设定和篇幅要求；填写完成后，点击页面右下角“编辑角色”，进入下一步。

第二步，编辑角色设定与角色关系。用户先在左侧角色档案卡中填写角色信息，再在右侧关系图中通过拖拽连线创建人物关系。保存后，如果觉得人物网络还不够完整，可以点击“AI补充”，由 Neuro AI 在不覆盖已有关系的前提下补全必要关系；完成后点击“生成大纲”进入下一步。

第三步，生成并调整大纲。Neuro AI 会根据基本信息、角色设定和人物关系生成四段式大纲与逐章细纲。用户可以直接修改章节范围，也可以输入整体反馈让 AI 重生成大纲；当大纲满意后，点击“生成正文”进入下一步。

第四步，生成人机协同正文。Neuro AI 会按章节逐步生成正文，用户可在生成过程中随时点击“停止生成”，稍后继续或放弃；生成完成后，用户既可以选中文本让 AI 局部重写，也可以直接手动编辑，并将喜欢的句子加入收藏。最终，用户可按需导出设定、大纲、单章正文、全书正文或全部内容。
